Overlap Diagnostic — usage walkthrough & template

A hands-on guide to code/diagnostic_protocol.py, the executable version of the paper's Section 6.1 protocol. Two worked examples with the paper's data, the exact console output (each step ends with a plain-language READING, and the run closes with a BOTTOM LINE), and a blank template you can copy for your own study.

Idea in one line. Before you extrapolate a threshold-local (RDD) effect to a whole population with a covariate model, let the first-stage propensity tell you whether the population is on support. Ample overlap → report the population effect. Weak overlap → don't; report an overlap-supported estimand and say for whom it holds.

What the tool does

  1. Step 1 — first stage. Cross-fit the propensity e(Z) and outcome models m(Z), μ₀(Z), μ₁(Z) (out-of-fold; folds clustered if you give a cluster id).
  2. Step 2 — overlap. Read the diagnostic off e(Z): the propensity AUC and maximum propensity (both base-rate-robust). Ends with a READING that says whether treated and control overlap.
  3. Step 3 — branch. AMPLE → report the population ATE and a CATE map. WEAK → refuse the population ATE; report ATO / trimmed / AutoDML and name the subpopulation.
  4. Step 4 — benchmark (optional). If a running variable is supplied, compare to the local RDD LATE and its confidence interval; the READING states whether the RDD confirms, rejects, or is too imprecise to judge the covariate estimate, and whether the near-threshold sample is representative.

Quick start

$ pip install numpy pandas scikit-learn statsmodels matplotlib econml scipy
$ cd replication/code
$ python diagnostic_protocol.py --app covenant # built-in example 1
$ python diagnostic_protocol.py --app meyersson # built-in example 2
$ python diagnostic_protocol.py # your own data via the CONFIG block

Example 1  ·  Covenant default   verdict: AMPLE overlap

Does technical default on a debt covenant lower investment, and can we extend the threshold-local effect to the whole cross-section? Configure the columns:

CONFIG = dict(
    data        = "../data/NWallnoloan_forest_data.csv",
    outcome     = "investment",       # Y : investment / capital
    treatment   = "bind",             # W : 1 = technical default on a covenant
    covariates  = ["lag.macro.q","cashflow","lag.cashflow","lag.log.assets",
                   "lag.altman_alt","initial_slack_ratio","syndicate_size",
                   "has.rating.all","cash_assets","year","quarter","firm_fe"],
                                       # Z : fundamentals, EXCLUDING the running variable
    running_var = "slack.ratio",       # optional : the RDD running variable
    cutoff      = 0.0,
    treat_when  = "below",             # W = 1 when slack < cutoff
    cluster     = "gvkey",             # optional : cluster id for standard errors
    outcome_scale = 100.0,             # report in percentage points
    label       = "Covenant default and investment",
)
$ python diagnostic_protocol.py --app covenant

Example 2  ·  Meyersson close elections   verdict: WEAK overlap

Does a narrowly-elected Islamic mayor raise female schooling? Prior Islamic support almost perfectly predicts who wins, so overlap is at risk. Configure the columns:

CONFIG = dict(
    data        = "../data/meyersson_clean.csv",
    outcome     = "hischshr1520f",     # Y : female high-school share
    treatment   = "i94",              # W : 1 = Islamic mayor won 1994
    covariates  = ["vshr_islam1994","partycount","shhs","lpop1994","ageshr19",
                   "ageshr60","sexr","merkezi","merkezp","subbuyuk","buyuk",
                   "hischshr1520m"],   # Z : municipal covariates, EXCLUDING the margin
    running_var = "iwm94",             # optional : Islamic win margin (RDD running variable)
    cutoff      = 0.0,
    treat_when  = "above",             # W = 1 when margin >= cutoff
    cluster     = "pcode",             # optional : province cluster id
    outcome_scale = 100.0,
    label       = "Islamic mayor and female schooling (Meyersson 2014)",
)

Bottom line: do not report a population effect; report an overlap-supported estimand and name the subpopulation.

Example 3  ·  A clean RDD (synthetic demo)   verdict: AMPLE & the RDD validates

Neither paper example reaches the happy case where the RDD actually validates the covariate CATE (covenant's RDD is too imprecise; Meyersson's overlap fails). This synthetic dataset is built so all three conditions hold at once — ample overlap, a precise RDD, and selection-on-observables — so the design confirms the covariate extrapolation. Generate it with python make_demo_rdd.py, then:

CONFIG preset "demo":  data = demo_rdd_clean.csv,  outcome = "y",  treatment = "w",
                       covariates = ["x1","x2","x3"],  running_var = "r",  cutoff = 0,  treat_when = "above"

Aside — can you rescue Meyersson by choosing a smaller Z? No. Dropping the covariate that kills overlap (prior Islamic vote share) does make overlap ample (AUC 0.97 → 0.71), but that covariate is the key confounder: without it the estimate turns to −1.4pp, the wrong sign versus the RDD (+3.0pp). The covariate you need for identification is exactly the one that destroys overlap — a genuine trade-off, reproduced by demo_Z_choice.py.

Use it on your own data  ·  template

Copy this CONFIG into diagnostic_protocol.py (replacing the block near the top), fill in your column names, and run python diagnostic_protocol.py. The only hard rule: the running variable must be excluded from covariates.

CONFIG = dict(
    data        = "PATH/TO/your_data.csv",
    outcome     = "YOUR_OUTCOME_COL",       # Y  (numeric)
    treatment   = "YOUR_TREATMENT_COL",     # W  (0 / 1)
    covariates  = ["col1","col2", "..."],   # Z  : pre-treatment covariates.
                                            #      >>> EXCLUDE the running variable <<<
    running_var = "YOUR_RUNNING_VAR",       # optional : omit/None if not an RDD
    cutoff      = 0.0,                       # threshold c
    treat_when  = "below",                   # "below" if W=1 when R<c, else "above"
    cluster     = "YOUR_CLUSTER_ID",         # optional : id for clustered SEs (or None)
    outcome_scale = 1.0,                     # multiply Y for reporting units
    label       = "My application",
)

Field reference

FieldWhat to put
dataPath to a CSV (one row per unit).
outcome / treatmentColumn names for Y (numeric) and W (coded 0/1).
covariatesPre-treatment covariates Z. Exclude the running variable.
running_var, cutoff, treat_when Optional RDD benchmark. treat_when="below" if W=1 when R<cutoff, else "above". Omit running_var for a purely observational study.
clusterOptional id for clustered standard errors and clustered cross-fitting folds.
outcome_scaleMultiplier for reporting units (e.g. 100 to report percentage points).

What you end up estimating

VerdictWhat to reportWhat NOT to report
AMPLE Population ATE (doubly-robust, clustered SE) and the CATE map τ(z); the extension of the local effect is supported. —
WEAK An overlap-supported estimand — ATO, trimmed ATE, or AutoDML — with the subpopulation named; plus the statement that the population effect is not identified from the observed support. The all-population ATE (it extrapolates off-support and can flip sign).

Decision rule (adjustable at the top of the script): overlap is treated as ample when propensity AUC < 0.95 and max propensity < 0.98; otherwise weak. Each run also writes diagnostic_overlap.pdf (the treated-vs-control propensity histogram) and diagnostic_report.csv (the machine-readable summary).
